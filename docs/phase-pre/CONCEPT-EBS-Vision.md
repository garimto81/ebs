# EBS 비전 문서

이 문서는 EBS 프로젝트의 배경, 목표, 전략을 설명합니다. EBS가 단순한 RFID 소프트웨어가 아니라, **포커 방송 프로덕션 전체 워크플로우의 인프라스트럭처**임을 명확히 합니다.

**이 문서는 기획/비전 문서입니다. 기술적 구현 방법은 별도 설계 문서를 참조하세요.**

---

## 1. 왜 EBS를 만드는가?

### 1.1 현재 상황

**PokerGFX 의존도 100%**

현재 모든 포커 방송은 PokerGFX 소프트웨어에 100% 의존하고 있습니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                       현재 시스템 흐름                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  RFID Table ──▶ PokerGFX Server ──▶ BRACELET STUDIO ──▶ 방송 송출 │
│                       │                                             │
│                       ▼                                             │
│                  PokerGFX DB                                        │
│                  (외부 소유)                                        │
│                       │                                             │
│                       ▼                                             │
│              핸드 히스토리, 플레이어 통계                            │
│                       │                                             │
│                       ▼                                             │
│              JSON 파일로 추출 (수작업)                               │
│                       │                                             │
│                       ▼                                             │
│              분석 작업 (많은 시간/인원 투입)                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

문제점:
├── 모든 데이터가 외부 시스템에 저장
├── 데이터 추출에 수작업 필요
├── 분석에 많은 인원과 시간이 투입됨
└── 라이선스 만료 시 방송 중단
```

### 1.2 핵심 문제점

| 문제 | 설명 | 영향 |
|------|------|------|
| **공급업체 종속** | PokerGFX 서비스 중단 시 대안 없음 | 방송 완전 중단 위험 |
| **데이터 비소유** | 핸드 히스토리, 플레이어 통계가 외부 소유 | 자유로운 데이터 활용 불가 |
| **수작업 병목** | 자막 연출에 많은 수작업 필요 | 운영 인력 과다 투입 |
| **기능 제약** | 커스터마이징 불가, 요청 기능 추가 어려움 | 차별화된 방송 불가 |
| **비용 누적** | 연간 라이선스 비용 지속 발생 | 장기적 비용 부담 |

### 1.3 EBS가 해결하는 것

| 문제 | EBS 해결책 | 기대 효과 |
|------|-----------|----------|
| 공급업체 종속 | 자체 시스템 소유 및 운영 | **자산 내재화/독립화** |
| 데이터 비소유 | 자체 데이터 저장, 완전한 소유권 | 무제한 데이터 활용 |
| 수작업 병목 | 자막 연출 워크플로우 자동화 | 인원/시간 대폭 절감 |
| 기능 제약 | 자체 개발로 무한 확장 | 차별화된 방송 콘텐츠 |
| 비용 누적 | 초기 개발 후 무비용 운영 | 장기 비용 절감 |

## 2. EBS의 역할: 프로덕션 인프라스트럭처

### 2.1 EBS는 단순 소프트웨어가 아니다

**EBS = 포커 방송 프로덕션 전체 워크플로우의 인프라**

```
┌─────────────────────────────────────────────────────────────────────┐
│                 포커 방송 프로덕션 전체 구조                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    방송 1회 운영 인력                         │   │
│  │                                                             │   │
│  │   [현장]              [중계]              [후반]            │   │
│  │   ────────           ────────           ────────           │   │
│  │   테이블 운영         카메라 감독         편집              │   │
│  │   딜러               스위처             자막 제작          │   │
│  │   플로어             오디오             통계 분석          │   │
│  │   RF 기술            조명               하이라이트         │   │
│  │   ...               GFX 운영           업로드             │   │
│  │                     ...               ...                │   │
│  │                                                             │   │
│  │   총 인원: 약 30명                                          │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                      │
│                              ▼                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                       EBS 자동화 영역                        │   │
│  │                                                             │   │
│  │   • 자막 연출 자동화 (수작업 제거)                           │   │
│  │   • 플레이어 정보 자동 입력                                 │   │
│  │   • 핸드 히스토리 자동 생성                                 │   │
│  │   • 통계 자동 계산                                         │   │
│  │   • 하이라이트 장면 자동 감지                               │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                      │
│                              ▼                                      │
│                                                                     │
│              목표: 30명 → 15~20명으로 축소 가능                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 자막 연출 워크플로우의 문제

현재 포커 방송의 자막 연출은 **엄청난 수작업**을 필요로 합니다:

```
현재 자막 연출 워크플로우 (수작업 중심)
──────────────────────────────────────────────────────────────────

1. 플레이어 등장
   └── GFX 운영자가 수동으로 이름, 스택, 포지션 입력
   └── 오타 발생 시 수정 (실시간 스트레스)

2. 카드 공개
   └── RFID로 카드 인식
   └── 그러나 승률, 통계는 별도 입력 필요

3. 베팅 액션
   └── GFX 운영자가 모든 액션 실시간 입력
   └── 폴드/콜/레이즈/올인 수동 트리거
   └── 팟 사이즈 수동 계산

4. 핸드 종료
   └── 결과 수동 기록
   └── 핸드 히스토리 수동 저장
   └── JSON 추출 후 별도 분석

5. 방송 후
   └── 통계 수동 집계
   └── 하이라이트 장면 수동 선별
   └── 편집팀에 수동 전달

문제:
├── GFX 운영자 최소 2명 상시 필요
├── 분석 인력 별도 필요
├── 실시간 입력 오류 빈번
├── 방송 후 추가 작업 과다
└── 인력 비용 과다
```

## 3. EBS 핵심 가치

### 3.1 자산 내재화/독립화 (최우선)

**"우리 시스템은 우리가 소유한다"**

EBS의 가장 중요한 목표는 **자산 내재화와 독립화**입니다.

```
현재 상황:
├── 방송 시스템이 외부 업체에 완전 의존
├── 계약 종료 시 시스템 사용 불가
├── 데이터 접근 및 활용 제약
└── 기능 개선/확장에 외부 협조 필요

EBS 목표:
├── 시스템 전체를 자체 소유
├── 영구적인 사용권 확보
├── 모든 데이터에 대한 완전한 통제권
└── 자유로운 기능 개선/확장
```

**자산 내재화의 의미:**

| 영역 | 현재 (PokerGFX) | EBS 목표 |
|------|----------------|----------|
| **시스템 소유** | 라이선스 임대 | 완전 소유 |
| **데이터 소유** | 외부 저장 | 자체 저장/소유 |
| **운영 통제** | 외부 의존 | 자체 통제 |
| **기능 확장** | 업체 협조 필요 | 자체 결정 |
| **장애 대응** | 외부 의존 | 자체 대응 |

### 3.2 운영 효율화

**"같은 결과물을 더 적은 인원으로, 같은 인원으로 더 많은 결과물을"**

자동화의 목표는 명확해야 합니다. 애매한 '자동화'라는 용어는 의미가 없습니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                     운영 효율화 두 가지 목표                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  목표 A: 동일 결과물 → 더 적은 인원                                  │
│  ─────────────────────────────────────                              │
│                                                                     │
│  현재: 방송 1회 운영에 약 30명 필요                                  │
│         (현장 10, 중계 12, 후반 8)                                  │
│                                                                     │
│  EBS:  방송 1회 운영에 15~20명으로 축소 가능                         │
│         • GFX 운영: 2명 → 1명 (자막 자동화)                         │
│         • 분석: 2명 → 0명 (통계 자동 생성)                          │
│         • 후반 편집 지원: 3명 → 1명 (자동 하이라이트)               │
│                                                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  목표 B: 동일 인원 → 더 많은 결과물                                  │
│  ─────────────────────────────────────                              │
│                                                                     │
│  현재 자막 연출 워크플로우:                                         │
│         • 모든 플레이어 정보 수동 입력                              │
│         • 모든 베팅 액션 수동 트리거                                │
│         • 통계 수동 계산                                           │
│         • 하이라이트 수동 선별                                     │
│                                                                     │
│  EBS 자동화 후:                                                     │
│         • 플레이어 정보 자동 로드                                   │
│         • 액션 자동 감지 및 자막 자동 트리거                        │
│         • 통계 실시간 자동 계산                                     │
│         • 하이라이트 장면 자동 마킹                                 │
│                                                                     │
│         → 동일 인원이 2배 이상의 콘텐츠 처리 가능                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**운영 효율화 측정 지표:**

| 영역 | 현재 | 목표 | 측정 방법 |
|------|:----:|:----:|----------|
| 방송 1회 필요 인원 | 30명 | 15~20명 | 운영 로그 |
| GFX 운영 인원 | 2명 | 1명 | 배치표 |
| 자막 입력 자동화율 | 0% | 80%+ | 수동/자동 비율 |
| 후반 작업 시간 | 4시간 | 1시간 | 작업 시간 |

## 4. 단계별 접근 (2단계 전략)

EBS는 두 단계로 진행됩니다:

```
┌─────────────────────────────────────────────────────────────────────┐
│                        EBS 2단계 전략                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1단계: PokerGFX 완전 동일 제품 개발                                 │
│  ─────────────────────────────────────                              │
│                                                                     │
│  목표: 기존 PokerGFX와 100% 동일한 기능의 자체 시스템                │
│                                                                     │
│  - 현재 방송 품질 그대로 유지                                       │
│  - 운영자가 새로 배울 것 없음                                       │
│  - 모든 기존 기능 동일하게 동작                                     │
│  - 차이점: 소유권이 우리에게 있음                                   │
│                                                                     │
│  ⚠️ 이 단계에서는 새로운 기능 추가 없음!                            │
│     순수하게 "동일 기능의 자체 시스템" 확보가 목표                   │
│                                                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  2단계: 운영 효율화 (자동화)                                         │
│  ─────────────────────────────                                      │
│                                                                     │
│  1단계 완료 후에만 논의                                             │
│                                                                     │
│  - 목표 A: 동일 결과물, 더 적은 인원 (30명 → 15~20명)               │
│  - 목표 B: 동일 인원, 더 많은 결과물 (자막 연출 자동화)             │
│                                                                     │
│  ⚠️ 1단계가 완료되어야 2단계 상세 기획 시작                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**단계별 상세:**

| 단계 | 목표 | 핵심 질문 | 성공 기준 |
|:----:|------|----------|----------|
| **1단계** | PokerGFX 동일 제품 | "기존과 똑같이 동작하는가?" | 기존 방송 품질 유지 |
| **2단계** | 운영 효율화 | "더 적은 인원으로 가능한가?" | 인원 또는 산출물 개선 |

## 6. 하드웨어는 5%

### 6.1 프로젝트 비중

```
┌─────────────────────────────────────────────────────────────────────┐
│                      EBS 프로젝트 비중                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  소프트웨어 개발 (서버, 화면, 연동)                                  │
│  ████████████████████████████████████████████████████████ 60%      │
│                                                                     │
│  데이터 연동/통합 (기존 시스템 연계)                                 │
│  ██████████████████████████████ 25%                                │
│                                                                     │
│  운영 워크플로우 설계                                               │
│  ████████████ 10%                                                  │
│                                                                     │
│  RFID 하드웨어                                                      │
│  ██████ 5%                                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.2 하드웨어가 5%인 이유

```
하드웨어 구성:
├── 기성품 구매 (MCU, RFID 모듈)
├── 설계 외주 (PCB, 안테나)
└── 조립 외주 (업체에서 완제품 납품)

우리가 직접 하는 것:
├── 요구사항 정의 ("이런 기능 필요합니다")
├── 스펙 검증 ("제대로 동작하는지 확인")
└── 테스트 ("실제 방송 환경에서 테스트")

→ 하드웨어는 '구매'와 '검증'이 주된 작업
→ 직접 개발하는 영역은 소프트웨어
```
