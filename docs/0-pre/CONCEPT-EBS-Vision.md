## Executive Summary

> **"왜 우리는 GFX를 대체해야 하는가?"**

EBS 프로젝트의 핵심은 **RFID 보드 제작이 아닙니다**. 보드는 전체 프로젝트의 5%도 되지 않습니다.

**진짜 핵심**:
1. WSOP STUDIO에서 축적된 **데이터 자산을 내재화**
2. 외부 솔루션(PokerGFX) 의존에서 **완전한 독립**
3. 방송 워크플로우의 **80% 자동화**

본 문서는 "왜 해야 하는가"와 "무엇을 해야 하는가"에 대한 **개념적 정의**를 제공합니다.

### Infrastructure Perspective

EBS는 단순한 "RFID 보드 프로젝트"가 아닙니다.

**EBS = 포커 프로덕션의 데이터 인프라스트럭처**

## Part 1: 왜 GFX를 대체해야 하는가?

### 1.1 PokerGFX 종속의 위험

현재 WSOP STUDIO 방송은 PokerGFX에 **100% 의존**합니다.

| 위험 요소 | 현재 상태 | 비즈니스 영향 |
|----------|----------|--------------|
| **라이선스 비용** | 연간 $XX,XXX | 지속적 비용 증가 위험 |
| **기술 지원** | 평균 72시간 대기 | 방송 중단 시 즉각 대응 불가 |
| **데이터 접근** | 제한적 Export | 핸드 히스토리 활용 제한 |
| **커스터마이징** | 제공 스킨만 사용 | 브랜드 차별화 불가 |
| **서비스 종료** | 언제든 가능 | 대안 없이 방송 중단 |

### 1.2 GFX가 없어진다면?

**시나리오**: PokerGFX가 서비스를 종료하거나, 라이선스 비용이 감당 불가능해진다면?

| 영향 범위 | 즉각적 결과 | 복구 시간 |
|----------|------------|----------|
| **라이브 방송** | 카드 그래픽 표시 불가 | - |
| **시청자 경험** | 핸드 정보 없이 시청 | - |
| **데이터 축적** | 핸드 히스토리 수집 중단 | - |
| **대체 솔루션** | 현재 없음 | **12개월+** |

> **결론**: GFX 없이는 WSOP STUDIO 방송이 **불가능**합니다.
> 대안을 **지금** 준비하지 않으면, 비상 상황 발생 시 **최소 1년간 방송 불가**.

### 1.3 EBS가 해결하는 문제

| 문제 | GFX 현황 | EBS 솔루션 |
|------|----------|-----------|
| **종속성** | 외부 벤더 100% 의존 | 자체 시스템 완전 소유 |
| **데이터** | 외부 DB, 제한적 접근 | 자체 DB, 무제한 활용 |
| **비용** | 연간 라이선스 비용 | 초기 개발 후 무비용 |
| **지원** | 글로벌 72시간 대기 | 로컬 즉각 대응 |
| **확장성** | 벤더 기능 대기 | 필요 기능 자체 개발 |

## Part 2: GFX에 있었으면 했던 기능들

### 2.1 Pain Points (현재 운영 불만)

#### 2.1.1 운영 복잡성

| Pain Point | 현재 GFX | 원하는 개선 |
|------------|----------|------------|
| **전담 인력 필수** | 전문 오퍼레이터 필요 | 누구나 운영 가능한 UI |
| **모든 액션 수동** | Fold/Call/Raise 모두 입력 | 스마트 예측 + 원클릭 |
| **칩 카운트 수동** | 브레이크마다 수동 입력 | WSOP+에서 자동 동기화 |
| **실수 복구 어려움** | Undo 기능 제한적 | 무제한 히스토리 + Undo |

#### 2.1.2 자동화 부족

| Pain Point | 현재 GFX | 원하는 개선 |
|------------|----------|------------|
| **핸드 종료 수동** | 오퍼레이터 클릭 | 칩 이동 감지 자동 종료 |
| **하이라이트 수동** | PD가 수동 태깅 | 빅팟/블러프 자동 감지 |
| **플레이어 정보** | 매번 수동 입력 | CSV Import / API 연동 |
| **딜러 버튼 이동** | 수동 조작 | 자동 이동 |

#### 2.1.3 데이터 활용 제한

| Pain Point | 현재 GFX | 원하는 개선 |
|------------|----------|------------|
| **데이터 Export 제한** | 특정 포맷만 | SQL 직접 쿼리, 다양한 포맷 |
| **통계 항목 고정** | VPIP, PFR만 | 커스텀 통계 무제한 |
| **콘텐츠 수동 제작** | 하이라이트 수동 편집 | 자동 클립 생성 |
| **분석 불가** | Raw 데이터 접근 불가 | Jupyter 노트북 연동 |

#### 2.1.4 시청자 경험 제한

| Pain Point | 현재 GFX | 원하는 개선 |
|------------|----------|------------|
| **통계 정보 부족** | 기본 스탯만 | 상세 플레이어 프로필 |
| **히스토리 재생** | 불가 | 과거 핸드 검색/재생 |
| **인터랙티브 제한** | 정적 오버레이 | 시청자 선택형 뷰 (미래) |

## Part 3: WSOP+ 데이터 활용 방안

### 3.0 중요 제약사항 (2026-01-23 확인)

> ⚠️ **PokerGFX Live Hand API = Enterprise 라이선스 전용**
>
> PokerGFX support 회신 (2026-01-23):
> "The PokerGFX live hand API (a json file that updates at the completion of each hand) is **no longer available by request**. This feature is now **only available with an Enterprise license**."

**영향 분석:**

| 시나리오 | 내용 | 대응 |
|----------|------|------|
| **Plan A** | Enterprise 라이선스 구매 | 비용 확인 필요 (추정 $50K+/년) |
| **Plan B** | 자체 데이터 수집 (RFID 기반) | EBS 핵심 가치, Stage 0 목표 |
| **Plan C** | CSV/수동 Import | 최소 기능, 자동화 불가 |

> **결론**: WSOP+ API 연동은 **불확실**합니다. EBS는 **Plan B (자체 데이터 수집)**를 기본 전략으로 진행합니다.

## Part 4: EBS의 3가지 핵심 가치

### 4.1 가치 1: 데이터 독립 (Data Independence)

> **"우리 데이터는 우리가 소유한다"**

| 현재 (GFX) | 미래 (EBS) |
|-----------|-----------|
| GFX DB에 종속 | 자체 DB 완전 소유 |
| Export 제한 | SQL 직접 쿼리 |
| 특정 포맷만 | JSON, CSV, Parquet 지원 |
| API 없음 | RESTful API + WebSocket |

### 4.2 가치 2: 운영 자동화 (Operational Automation)

> **"사람이 하는 일을 최소화한다"**

| 영역 | 현재 수동 작업 | EBS 자동화 | 절감 |
|------|--------------|-----------|------|
| 액션 입력 | 모든 액션 수동 | 스마트 예측 | 80% |
| 칩 카운트 | 브레이크 수동 | WSOP+ 동기화 | 100% |
| 하이라이트 | PD 수동 태깅 | 자동 감지 | 90% |
| 플레이어 설정 | 매 이벤트 수동 | CSV Import | 95% |

### 4.3 가치 3: 시청자 경험 (Viewer Experience)

> **"시청자에게 더 많은 정보를"**

| 정보 유형 | 현재 (GFX) | 미래 (EBS) |
|----------|-----------|-----------|
| 기본 스탯 | VPIP, PFR | + 3Bet, Fold to 3Bet, C-Bet |
| 플레이어 정보 | 이름, 스택 | + 국적, 수상 내역, 커리어 |
| 핸드 히스토리 | 없음 | 실시간 검색 가능 |
| 확률 정보 | 기본 Equity | + Outs, Odds, 시나리오별 확률 |

### 4.4 가치 4: 인프라 신뢰성 (Infrastructure Reliability)

> **"방송 전체가 EBS에 의존할 수 있다"**

| 지표 | 목표 |
|------|------|
| 가용성 | 99.9% (월 다운타임 < 43.8분) |
| 복구 시간 | 5분 이내 |
| 데이터 손실 | 최대 1핸드 |
| 동시 테이블 | 10개 |

EBS는 "있으면 좋은" 시스템이 아닙니다.
EBS가 멈추면 **방송 전체가 멈춥니다**.
따라서 인프라 수준의 신뢰성이 필수입니다.

## Part 7: 결론

### 7.1 왜 지금 시작해야 하는가?

1. **GFX 종속 위험**: 언제든 서비스 중단 가능
2. **데이터 축적**: 시작이 빠를수록 데이터 가치 증가
3. **기술 역량**: 하드웨어 경험 축적 기회
4. **비용 절감**: 장기적 라이선스 비용 제거

### 7.2 하드웨어는 5%

| 영역 | 비중 | 설명 |
|------|:----:|------|
| **RFID 하드웨어** | 5% | ESP32 + RFID 모듈 (이미 준비 완료) |
| **소프트웨어** | 60% | Viewer Overlay, Action Tracker, Console |
| **데이터 연동** | 25% | WSOP+ API, 핸드 히스토리, 통계 |
| **운영 워크플로우** | 10% | 교육, SOP, 병행 운영 |

> **보드 제작은 가장 쉬운 부분입니다.**
> **진짜 도전은 소프트웨어와 데이터입니다.**
