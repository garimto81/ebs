# EBS Slack 관리 시스템 (Slack Management System)

**작성일**: 2026-02-02
**대상**: EBS 프로젝트 팀 내부 커뮤니케이션
**버전**: 1.0.0

---

## 1. 개요

EBS 프로젝트의 팀 내부 커뮤니케이션을 체계적으로 관리하기 위한 Slack 운영 시스템입니다.

### 1.1 목적

| 목적 | 설명 |
|------|------|
| **지식 축적** | 기술 결정, 문제 해결 과정을 검색 가능한 형태로 보존 |
| **실시간 소통** | 빠른 질문-답변, 진행 상황 공유 |
| **문서 동기화** | 중요 결정사항을 `docs/decisions/`로 아카이빙 |
| **효율적 협업** | 채널별 목적 명확화로 정보 과부하 방지 |

### 1.2 설계 원칙

| 원칙 | 설명 | 적용 방식 |
|------|------|----------|
| **5-7개 채널 제한** | 채널 파편화 방지 | Stage별 + 주제별 혼합 구조 |
| **이모지 분류 시스템** | 메시지 성격 즉시 파악 | 리액션 이모지 코드 정의 |
| **스레드 활용 강제** | 메인 채널 깔끔 유지 | 모든 답변은 스레드로 |
| **아카이빙 자동화** | 결정사항 문서화 | Slack → Markdown 전환 |

## 2. 채널 구조

### 2.1 채널 목록 (총 6개)

```
EBS Workspace/
├── #ebs-general              (일반 공지/잡담)
├── #ebs-stage-current        (현재 Stage 작업)
├── #ebs-tech-discussion      (기술 논의)
├── #ebs-decisions            (결정사항 기록 전용)
├── #ebs-questions            (Q&A)
└── #ebs-external             (외부 업체 소통)
```

### 2.2 채널별 상세 설명

#### `#ebs-general` (일반)

| 항목 | 내용 |
|------|------|
| **목적** | 프로젝트 공지, 팀 빌딩, 비기술 논의 |
| **주 사용자** | 전체 팀 |
| **메시지 예시** | "Stage 0 킥오프 미팅 금요일 3시", "ESP32 보드 도착했습니다" |
| **규칙** | 📢 공지는 반드시 📌 고정 |

#### `#ebs-stage-current` (현재 Stage 작업)

| 항목 | 내용 |
|------|------|
| **목적** | 현재 진행 중인 Stage의 실시간 작업 내용 공유 |
| **주 사용자** | 개발자, PM |
| **메시지 예시** | "MFRC522 배선 완료, 테스트 중", "Serial 통신 JSON 파싱 이슈" |
| **규칙** | Stage 전환 시 채널명 변경 (`#ebs-stage-0` → `#ebs-stage-1`) |

**Stage 전환 프로세스:**
```
Stage 0 → Stage 1 Gate 통과 시:
1. #ebs-stage-current를 #ebs-stage-0으로 rename
2. #ebs-stage-0을 Archive
3. 새로운 #ebs-stage-current 생성
4. 채널 토픽: "Stage 1: PokerGFX 복제 (2026 H2)"
```

#### `#ebs-tech-discussion` (기술 논의)

| 항목 | 내용 |
|------|------|
| **목적** | 아키텍처, 기술 선택, 성능 최적화 논의 |
| **주 사용자** | 개발자, 아키텍트 |
| **메시지 예시** | "WebSocket vs SSE 어떤 게 나을까요?", "DB 스키마 리뷰 부탁드립니다" |
| **규칙** | 🧠 기술 결정이 완료되면 반드시 `#ebs-decisions`로 요약 이동 |

#### `#ebs-decisions` (결정사항 기록 전용)

| 항목 | 내용 |
|------|------|
| **목적** | 기술/비즈니스 결정사항 공식 기록 (검색 및 아카이빙 최적화) |
| **주 사용자** | PM, 아키텍트 (글쓰기 제한) |
| **메시지 예시** | "✅ 결정: ESP32 대신 Arduino 사용 (이유: ...)", "✅ 결정: Stage 0 기간 5개월로 연장" |
| **규칙** | ✅ 이모지로 시작, 매주 금요일 `docs/decisions/`로 아카이빙 |

**메시지 포맷:**
```markdown
✅ 결정: [제목]
날짜: 2026-02-02
결정자: [이름]

배경:
[왜 이 결정이 필요했는가]

결정 내용:
[무엇을 결정했는가]

근거:
[어떤 근거로 결정했는가]

대안:
[고려했지만 기각된 옵션]

영향:
[이 결정이 프로젝트에 미치는 영향]
```

#### `#ebs-questions` (Q&A)

| 항목 | 내용 |
|------|------|
| **목적** | 빠른 질문-답변 (Stack Overflow 스타일) |
| **주 사용자** | 전체 팀 |
| **메시지 예시** | "ESP32에서 Serial.println이 안 보이는데 baud rate 어떻게 확인하나요?", "Mifare Ultralight와 NTAG213 차이가 뭔가요?" |
| **규칙** | 해결 완료 시 ✅ 리액션 필수, 답변은 스레드로만 |

#### `#ebs-external` (외부 업체 소통)

| 항목 | 내용 |
|------|------|
| **목적** | 외부 업체(RFID 공급사, PCB 제조사) 관련 논의 |
| **주 사용자** | PM, 구매 담당자 |
| **메시지 예시** | "RFIDup에서 견적 왔습니다 (링크)", "JLCPCB 샘플 배송 추적 번호" |
| **규칙** | 메일 내용은 요약만, 원문은 `MAIL-MANAGEMENT.md` Tracker 참조 |

## 3. 메시지 분류 규칙

### 3.1 이모지 리액션 코드

**메시지 성격 분류** (메시지 작성자가 추가):

| 이모지 | 성격 | 사용 예시 |
|--------|------|----------|
| 📢 | 공지 | "내일 미팅 일정 변경되었습니다" |
| 🔧 | 기술 작업 | "MFRC522 라이브러리 업데이트 완료" |
| 🐛 | 버그 리포트 | "Serial 통신이 간헐적으로 끊김" |
| 💡 | 아이디어/제안 | "칩 카운트 자동 추적 기능 추가하면 어떨까요?" |
| ❓ | 질문 | "ESP32 플래시 모드 설정 어떻게 하나요?" |
| 📊 | 진행 상황 | "Stage 0 Week 3 완료, Gate 조건 70% 달성" |
| 🚨 | 긴급/차단 | "프로덕션 서버 다운, 즉시 확인 필요" |

**메시지 상태 분류** (다른 팀원이 추가):

| 이모지 | 상태 | 사용 예시 |
|--------|------|----------|
| ✅ | 해결/완료 | 질문 답변 완료, 버그 수정 완료 |
| 👀 | 검토 중 | 아키텍트가 기술 제안 검토 중 |
| 🔄 | 진행 중 | 이슈 해결 진행 중 |
| ⏸️ | 보류 | 외부 요인으로 대기 중 |
| 🚫 | 기각 | 제안 채택되지 않음 |
| 📝 | 아카이빙 필요 | 이 내용은 문서화 필요 |

### 3.2 스레드 사용 규칙

**스레드 필수 상황:**

| 상황 | 규칙 |
|------|------|
| 질문에 답변 | 답변은 100% 스레드로 (메인 채널 깔끔 유지) |
| 기술 논의 | 2개 이상 메시지 교환 시 스레드 전환 |
| 긴 설명 | 3줄 이상 설명은 스레드로 이동 |
| 코드 공유 | 코드 블록은 스레드에서만 |

**메인 채널 허용:**

| 상황 | 예시 |
|------|------|
| 공지 | "금요일 3시 미팅" |
| 한 줄 상태 | "RFID 테스트 완료" |
| 링크 공유 | "PRD 업데이트 (링크)" |

**스레드 제목 규칙:**

```
# 좋은 예
질문: "ESP32 Serial.begin(9600) 후에도 데이터 안 보임"
  └─ 답변: "baud rate 115200으로 변경하세요 (ESP32 기본값)"
  └─ 질문자: "✅ 해결됐습니다, 감사합니다!"

# 나쁜 예 (메인 채널에 3개 메시지)
[Alice] "ESP32 Serial 안 보여요"
[Bob] "baud rate 확인했어요?"
[Alice] "아 115200으로 하니까 되네요 감사합니다"
```

### 3.3 메시지 작성 가이드

**좋은 메시지 예시:**

```
🔧 MFRC522 카드 인식 구현 완료

완료 항목:
- ESP32 SPI 연결 (GPIO5 = SDA)
- MFRC522 라이브러리 통합
- UID 읽기 테스트 (5장 카드 100% 인식)

다음 단계:
- Serial JSON 포맷 정의
- Python 서버 연결

참조:
- 코드: firmware/rfid_reader/rfid_reader.ino
- 테스트 로그: (스레드 참조)
```

**피해야 할 메시지 예시:**

```
❌ "됐어요" (무엇이 됐는지 불명확)
❌ "이거 어떻게 해요?" (코드 붙여넣기 없음)
❌ "아 왜 안 되지..." (문제 설명 없음)
```

## 4. 아카이빙 워크플로우

### 4.1 대상 메시지 선별

**아카이빙 필요 조건** (하나 이상 만족):

| 조건 | 예시 |
|------|------|
| 📝 리액션 3개 이상 | 중요하다고 팀원들이 인식 |
| `#ebs-decisions` 채널 메시지 | 모든 결정사항 |
| 기술 결정 완료 | "WebSocket 채택" |
| 버그 해결 완료 (✅) | 동일 이슈 재발 방지용 |
| 외부 업체 합의 | 견적 승인, 계약 조건 |

**아카이빙 불필요:**

| 유형 | 이유 |
|------|------|
| 일상 대화 | "점심 뭐 먹어요?" |
| 일시적 상태 | "지금 커피 마시러 갑니다" |
| 중복 정보 | 이미 문서화된 내용 |

### 4.2 아카이빙 프로세스

**담당**: PM 또는 지정된 팀원
**주기**: 매주 금요일 오후 (주간 회의 직후)

```
1. Slack 검색: 이번 주 `in:#ebs-decisions` OR `has::memo:` (📝 리액션)
   ↓
2. 메시지 내용 복사 (스레드 포함)
   ↓
3. Markdown 파일 생성: docs/decisions/YYYY-MM-DD-{제목}.md
   ↓
4. 메타데이터 추가 (날짜, 결정자, 채널 링크)
   ↓
5. Git 커밋: docs(ebs): Archive decision - [제목]
   ↓
6. Slack에 아카이빙 완료 알림 (파일 링크)
```

## 6. 외부 업체 소통 규칙

### 6.1 외부 업체 Slack 초대 정책

**원칙**: **외부 업체는 Slack 초대하지 않음** (보안 및 정보 통제)

| 소통 채널 | 용도 | 비고 |
|----------|------|------|
| **이메일** | 공식 견적, 계약, 기술 문의 | `MAIL-MANAGEMENT.md` 참조 |
| **전화/화상** | 긴급 이슈, 킥오프 미팅 | 회의록 Slack에 공유 |
| **Slack (내부)** | 업체 관련 논의 | `#ebs-external` 채널 |

### 6.2 외부 업체 관련 메시지 규칙

**`#ebs-external` 채널 사용법**:

```markdown
# 좋은 예
```
